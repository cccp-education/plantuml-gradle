== ✅ EPIC PLT-I18N : Internationalisation — 10 Langues (TERMINÉ S160)

*Découverte* : Session 148 — Cadrage i18n pour les ~150 strings hardcodées du plugin.
*Objectif* : Remplacer les strings en anglais hardcodé dans le code Kotlin (logger,
exceptions, descriptions de tâches) par un système d'i18n standard (ResourceBundle +
MessageFormat). Les prompts LLM restent en anglais (prompt engineering, pas i18n).
*Charge* : ~28 pts (~6-7 sessions)
*Priorité* : P1 — Plugin publié sur Maven Central mais monolingue (en). L'i18n est
le prochain cap majeur pour rendre le plugin utilisable internationalement.
*Dépend de* : `education.cccp:i18n-contracts:0.0.1` (MEM-3, workspace-bom) — artefact N0
fournissant `SupportedLanguage`, `LanguageCatalog`, `I18nConfig`, `MessageBundle`, `LocaleResolver`.
Actuellement en mavenLocal seulement (MEM-3.6 publication Central pending).

=== Différence vs Bakery BKY-I18N

Bakery = i18n de templates Thymeleaf (`#{key}`). Plantuml = i18n de code Kotlin
(logger, exceptions, descriptions). Approche : `java.util.ResourceBundle` +
`java.text.MessageFormat` pour les messages paramétrés, wrapper `PlantumlMessages`.

=== Architecture i18n

*Principe* : `ResourceBundle` standard Java + `MessageFormat` pour les messages
paramétrés. Chaque langue a un fichier `Messages_{code}.properties` dans
`src/main/resources/i18n/`. La langue active est déterminée par `language` dans
`PlantumlConfig`, résolu via la cascade ConfigMerger 4 couches
(CLI > env vars > YAML > gradle.properties > defaults).

[source]
----
PlantumlConfig.language ──► ConfigMerger (4 couches)
    │
    ▼
PlantumlMessages.forLanguage(code) ──► ResourceBundle.getBundle("i18n/Messages", locale)
    │
    ▼
PlantumlMessages.get("key")           → String simple
PlantumlMessages.format("key", args)  → String formaté (MessageFormat)
----

== 🔴 EPIC PLT-BOUNDARY : Translation Boundary — Frontière de traduction i18n

*Vision* : voir `.agents/I18N_BOUNDARY_VISION.adoc`
*Principe* : l'i18n sert la compréhension. Toute traduction qui réduit la
compréhension est un défaut. On traduit le contenant (étiquettes de
présentation), on préserve le contenu (identité sémantique). Le champ lexical
métier se traduit avec choix idiomatique (TRANSLATE / BORROW).
*Charge* : ~26 pts (~6-7 sessions)
*Priorité* : 🔴 HAUTE — l'i18n actuelle (EPIC PLT-I18N) a traduit les logs mais
pas le livrable (le diagramme). Le contenu du diagramme reste monolingue.
*Statut* : ✅ TERMINÉ — 8/8 US terminés (US-BND-0..7)
*Dépend de* : néant (nouveau domaine)

=== Contexte

EPIC PLT-I18N a livré l'infrastructure i18n (`PlantumlMessages`, `ResourceBundle`,
`ConfigMerger` 4 couches, 8 scénarios Cucumber PASS). Mais :
. Les 9 fichiers `Messages_*.properties` sont des copies conformes de
  `Messages_en.properties` (zéro traduction réelle — coquille vide).
. Le cœur métier (diagrammes générés) n'est pas couvert : chaînes hardcodées
  dans `KnowledgeGraphRenderer.kt` et `DiagramProcessor.kt`.
. Aucune frontière sémantique : aucun concept domaine pour distinguer
  traduisible (étiquettes d'UI) vs préservé (identifiants, savoir extrait).

=== Natures de texte et stratégies

[cols="1,3,2,2"]
|===
| Nature | Exemples | Décision | Stratégie

| 1. Présentation
| `"Classes"`, `"Files"`, `"Extracted"`, `"Empty Knowledge Graph"`
| Traduisible
| TRANSLATE (mot-à-mot via Messages_*.properties)

| 2. Identité sémantique
| `LlmService`, `calls`, `community_0`, `pgvector`, formules math
| Préservé
| PRESERVE (tel quel)

| 3. Champ lexical métier
| `pipeline`, `rollback`, `dependency injection`, `cache`
| Traduisible idiomatique
| TRANSLATE (si traduction consacrée) / BORROW (si emprunt consacré)
|===

=== Composants domaine

* `TranslationCategory` — enum : `PRESENTATION`, `SEMANTIC_IDENTITY`, `LEXICAL_FIELD`
* `TranslationStrategy` — enum : `TRANSLATE`, `BORROW`, `PRESERVE`
* `TranslatableText` — value object (texte + catégorie)
* `ClassifiedText` — résultat de classification
* `TextClassifier` — classifie un texte dans une `TranslationCategory`
* `IdiomaticGlossary` — registry terme → {langue → traduction + stratégie}
* `NonTranslatableTermRegistry` — termes métier client (REAC, AFNOR, FPA)
* `TranslationResolver` — applique la bonne stratégie selon catégorie + glossaire
