= Session 162 (2026-07-06) — EPIC 13 Cross-provider Fallback (6/6 tâches)

== Mission

Dernière tâche EPIC 13 (LOW) : fallback inter-provider quand un provider pool
est entièrement saturé. Exemple : Gemini Pro épuisé → Mistral Free → Ollama.
Méthodologie baby-step DDD/TDD/BDD (unit → functional → cucumber), agile
software craftsmanship, Red→Green→Refactor, 1 test à la fois.

== Fait

=== Étape A — TDD unit : `ApiKeyPool.isPoolSaturated()`

- `ApiKeyPoolSaturationTest.kt` créé (3 tests unitaires) — Red (Unresolved reference) → Green ✅
  - false quand aucune entry n'a dépassé son quota
  - true quand toutes les entries ont dépassé leur quota
  - false quand au moins une entry reste disponible
- `ApiKeyPool.isPoolSaturated()` implémenté dans `ApiKeyPool.kt` — 3 lignes logique
  - `entries.isEmpty() return true` (sécurité)
  - `entries.all { tracker.isQuotaExceeded(it) }`
- Concept DDD : brique de base pour le cross-provider fallback

=== Étape B — TDD unit : `CrossProviderFallbackOrchestrator`

- `CrossProviderFallbackOrchestratorTest.kt` créé (5 tests unitaires) — Red → Green ✅
  - Retourne une clé du 1er provider quand son pool n'est pas saturé
  - Fallback au provider suivant quand le 1er pool est saturé
  - Fallback à travers 2 providers saturés vers le 3e
  - Retourne null quand tous les pools sont saturés
  - Retourne null quand le fallback order ne contient aucun pool connu
- `CrossProviderFallbackOrchestrator.kt` créé (25 lignes, classe domaine pure)
  - `pools: Map<String, ApiKeyPool>`, `fallbackOrder: List<String>`
  - `selectKey()` : itère sur l'ordre, skip les pools saturés, retourne 1re clé dispo
- Concept DDD : service domaine au-dessus des pools par provider

=== Étape C — Functional : câblage dans LlmService

- `CrossProviderFallbackFunctionalTest.kt` créé (5 tests fonctionnels) — Green ✅
  - Gemini non saturé → clé Gemini retournée
  - Gemini saturé → fallback Mistral
  - Gemini + Mistral saturés → fallback Ollama
  - Tous saturés → null
  - Skip providers absents de la map
- `LlmService` modifié :
  - `fallbackOrchestrator: CrossProviderFallbackOrchestrator` ajouté (lateinit)
  - `initializeFallbackOrchestrator()` — ordre = provider actif + autres pools triés (stable)
  - `selectKeyWithCrossProviderFallback(provider)` — préfère le pool primaire, fallback orchestrateur si saturé
  - `getApiKeyFromPool()` refactorisé pour utiliser la nouvelle méthode
- Smart-cast Kotlin : `assertNotNull` JUnit5 ne propage pas → `!!` sur les assertions `assertEquals`

=== Étape D — BDD Cucumber

- `20_cross_provider_fallback.feature` créée (5 scénarios Gherkin taggés `@cross-provider`, EN only) ✅
  - Primary provider key returned when pool not saturated
  - Falls back to next provider when primary saturated
  - Falls back across two saturated providers to the third
  - Returns no key when every provider pool saturated
  - Skips providers absent from the pools map
- `CrossProviderFallbackSteps.kt` créé (12 steps Given/When/Then) ✅
- `PlantumlWorld` étendu : `crossProviderOrchestrator`, `crossProviderSelectedKeys` ✅
- Fix calibration Cucumber :

  - Quota threshold : `limit=4, threshold=50` (seuil=2) pour permettre 2 usages avant saturation
  - Scénario "all saturated" : 5 appels (2+2+1 null), pas 4
- Filtre `-Pcucumber.tags="@cross-provider"` → 5/5 scénarios PASS ✅
