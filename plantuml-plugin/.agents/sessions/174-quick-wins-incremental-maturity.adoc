== Résumé

Trois tâches accomplies en baby-step DDD/TDD/BDD :

=== QW-1 — validateDiagram() parsing réel (~30min)

**Problème** : `DiagramProcessor.validateDiagram()` (lignes 474-510) appelait
le LLM mais jetait la réponse, retournant toujours `score=8` hardcodé.

**Solution** :
- `ObjectMapper.readValue(validationResult, ValidationFeedback::class.java)` avec
  `KotlinModule` pour désérialiser la data class Kotlin.
- Fallback gracieux (score=5, feedback default) si parse échoue.
- Clé i18n `processor.validation_parse_failed` ajoutée aux 10 langues + default.
- Fix régression `DiagramProcessorPrivateMethodsTest` (assertion `isValid`/`valid`
  avec KotlinModule Jackson serialization).

**Tests** : 2 nouveaux tests Red→Green (`should parse LLM JSON`, `should fallback
on invalid JSON`). 9/9 `DiagramProcessorErrorHandlingTest` PASS.

=== QW-2 — api → implementation audit (~20min)

**Problème** : 8 dépendances en `api()` exposent ~20 transitives aux consommateurs.

**Migration** : `api` → `implementation` pour :
- `libs.bundles.plantuml` (plantuml-engine)
- `libs.bundles.jgit` (jgit-core, jgit-ssh, jgit-archive — non utilisés dans le code)
- `libs.commons.io` (non utilisé dans le code)
- `libs.bundles.plantuml.ai` (langchain4j × 9 modules)
- `libs.testcontainers.pg`
- `com.fasterxml.jackson.module:jackson-module-kotlin`
- `com.fasterxml.jackson.dataformat:jackson-dataformat-yaml`
- `com.fasterxml.jackson.datatype:jackson-datatype-jsr310`

**Justification** : l'API publique du plugin Gradle (`PlantumlExtension`,
`DefaultTask` subclasses) n'expose que des `Property<String>` Gradle standard.
Aucune lib externe n'est transmise aux consommateurs.

**Zéro régression** : `compileKotlin` OK, `test` OK (test préexistant
`nonexistent directory` échoue sur `main` aussi, non lié).
